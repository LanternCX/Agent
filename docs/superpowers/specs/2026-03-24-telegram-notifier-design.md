# 完成提醒 Telegram 通知 Skill 设计

## 背景

当前仓库已经开始积累 skill、plan 与规则，但还没有一个统一的全局通知能力。在某些任务执行完成后，如果用户已经离开终端，就无法第一时间知道任务已经结束。

用户最初提出过“任务完成时间超过 10 分钟后发送通知”的想法，但在讨论后确认：固定时长门槛过于机械，也会增加实现复杂度。之后又进一步确认，单纯依赖“长任务”这个说法，边界也不够清楚。更适合的方案是把判断口径改成：当 Agent 认为当前任务不是即时完成型工作、用户可能已经离开等待界面、且此刻已经真正完成时，主动调用一个全局复用的通知 skill，将完成消息转发到 Telegram 机器人。

## 目标

- 提供一个定位为全局复用的 skill，用于在任务完成后发送 Telegram 提醒
- 让 Agent 自主判断当前结果是否值得在完成后主动提醒用户，不依赖固定分钟阈值
- 用一个很小的本地脚本承担实际发送动作，避免把可靠性完全压在 skill 文本流程上
- 首次使用时由 Agent 引导完成初始化，不要求用户手动研究配置流程
- 发送失败不能影响主任务完成

## 非目标

本次设计不覆盖以下内容：

- 不做固定时长计时器，也不保留隐藏的分钟阈值
- 不做任务进行中的阶段性进度通知
- 不做多渠道通知扩展（如邮件、Slack、飞书）
- 不做复杂的任务历史、统计面板或消息归档系统
- 不依赖特定平台的 hook 机制作为核心实现

## 设计原则

### 1. 以 Tool Wrapper 为核心

这个 skill 的主体定位是 Tool Wrapper。skill 负责告诉 Agent 何时应该调用通知能力、调用前后需要满足什么约束；真正的外部动作由本地脚本完成。

### 2. Agent 判断是否值得提醒

判断标准不由时间阈值定义，也不再简单依赖“长任务”这一模糊说法，而由 Agent 结合上下文判断：这次完成是否值得主动提醒用户。这样更贴近真实工作流，也更符合通知能力本身的目的。

### 3. 渐进式引入

主入口文件只保留最核心规则。首次初始化细节和用户使用说明拆到独立文档中，避免把所有流程都塞进一个主 Markdown 文件里。

### 4. 流程尽量轻

虽然需要一个最小的完成后流程，但不把整个能力设计成沉重的机械流水线。只保留必要约束：任务真的完成、值得提醒、只发一次、失败不阻塞。

## 模式选择

本 skill 采用以下组合：

- 主模式：Tool Wrapper
- 辅模式：Generator

其中：

- Tool Wrapper 用于封装“发送 Telegram 通知”这个外部能力
- Generator 用于约束通知文案结构，避免不同任务下输出风格漂移过大

## 用户体验

用户使用这套 skill 时，无需单独管理计时器，也不需要记住额外的配置步骤。

当 Agent 判断当前任务不是即时完成型工作、用户可能在等待过程中离开，而任务现在已经真正完成时，Agent 额外调用这个 skill，向 Telegram 机器人发送一条完成提醒。用户即使暂时离开终端，也能收到“任务已完成”的消息。

如果是第一次需要发送通知，Agent 会先进入一次很短的初始化引导，收集 Telegram 所需的两个值，然后自动写入本地私有配置，再继续本次通知流程。

## 是否值得提醒的判断标准

skill 需要在 description 和正文中明确提示 Agent：以下情况通常说明“这次完成值得主动提醒用户”。

- 任务不是即时完成型回复，而是经历了一段需要等待的处理过程
- 任务包含多个阶段，例如搜索、设计、修改、验证、总结
- 任务虽然不大，但完成前已经让用户等待了一段时间，用户可能已经切走
- 任务完成这件事本身值得被单独提醒一次

以下情况通常不应触发通知：

- 简单问答或说明性回复
- 很快完成的小改动
- 仅执行单个短命令并立即返回结果
- 任务尚未真正完成，仍在等待用户确认或补充信息

其中，“是否值得提醒”优先于“是否看起来是长任务”。例如一个任务虽然不大，但如果已经让用户等待了 1 到 2 分钟、且结果值得在完成后提醒，那么也应视作可通知场景。

## 总体架构

建议目录结构如下：

- `skill/telegram-notifier/SKILL.md`
- `skill/telegram-notifier/README-zh.md`
- `skill/telegram-notifier/references/setup-telegram.md`
- `skill/telegram-notifier/scripts/send_telegram.py`
- `skill/telegram-notifier/assets/message-template.md`
- `skill/telegram-notifier/config.example.env`
- `skill/telegram-notifier/config.local.env`

整体职责拆分如下：

### `SKILL.md`

负责：

- 描述何时应该使用该 skill
- 明确“是否值得提醒”的判断标准
- 约束调用时机必须是任务真正完成之后
- 在发现未配置时转入初始化引导
- 指导 Agent 调用本地发送脚本
- 规定失败不阻塞主任务、不要重复发送

`SKILL.md` 不承担完整用户手册职责，也不展开写首次初始化的细节。

### `README-zh.md`

面向使用者，负责解释：

- 这是什么能力
- 为什么会收到这样的通知
- 第一次使用时会发生什么
- 本地配置保存在哪里
- 之后如何继续使用
- 遇到常见问题时如何理解

### `references/setup-telegram.md`

只负责首次初始化引导细节，内容包括：

- 需要向用户收集哪两个值
- 提问顺序
- 如果用户暂时不想配置应如何收尾
- 如何写入 `config.local.env`
- 写入完成后如何返回主流程

### `scripts/send_telegram.py`

负责：

- 读取本地配置
- 接收最终消息文本
- 调用 Telegram Bot API 发送消息
- 返回明确的成功或失败结果

这个脚本不负责判断是否值得提醒，也不负责决定是否发送；这些判断留在 skill 中完成。

### `assets/message-template.md`

负责定义统一的通知文案结构，避免每次调用都临时组织文本，导致通知内容不一致。

### `config.example.env`

提供最小配置模板，作为配置结构参考。

### `config.local.env`

这是实际运行时使用的本地私有配置文件，放在 skill 自己的目录下，由 Agent 在首次使用时引导创建，不应提交到版本控制。

## 配置设计

配置保持极简，只保留通知真正需要的信息。

建议使用 `.env` 风格配置，核心字段如下：

- `TELEGRAM_BOT_TOKEN`：Telegram 机器人令牌
- `TELEGRAM_CHAT_ID`：接收通知的聊天 ID

本次不加入固定分钟阈值配置，因为“是否值得提醒”完全由 Agent 判断。

## 配置文件位置

本次实现与验证仅基于当前仓库，不依赖用户主目录下的真实配置文件。

仓库内保留 `skill/telegram-notifier/config.example.env` 作为模板；真实运行配置使用 `skill/telegram-notifier/config.local.env`。这个文件不要求用户手动准备，而是在首次需要发送通知时，由 Agent 引导用户填写两个值并自动生成。

## 消息内容设计

通知内容应保持极简，因为它的目标不是汇报结果，而是把用户的注意力重新拉回当前 session。

Telegram 通知必须被定义为“提醒型消息”，而不是“报告型消息”。它只负责告诉用户：当前这次工作已经有结果，可以回来继续看会话内容；它不承担完整结果汇报职责。

建议消息包含以下信息：

- 项目名
- session 标识
- task 的极短标签
- 已完成这一事实
- 任务开始时间

消息示例可以是类似下面的风格：

“`[Agent][agent-repo]` Session abc123 已完成：Telegram notifier。开始于 14:02。”

消息应尽量保持单行，默认不写详细结果，不写长摘要，不写多条要点，不写技术细节。用户如果想看完整内容，应回到当前会话中查看。

## 脚本输入约定

发送脚本采用“接收整段最终消息文本”的方式，而不是接收一组分散的结构化参数。

这样约定的原因是：

- skill 已经负责基于模板组织好通知文案
- 脚本只需专注于发送，不再重复承担拼装消息的职责
- 后续如果模板微调，不需要同步改动脚本参数设计

因此脚本的职责边界进一步收敛为：读取配置、接收一段最终消息、调用 Telegram API、返回成功或失败结果。

## 调用流程

Agent 在一个值得提醒的任务完成后，调用这个 skill 的建议顺序如下：

1. 确认当前任务已经真正完成，而不是中途进展
2. 根据 skill 中的判断标准，确认这次完成值得发送提醒
3. 检查 `config.local.env` 是否存在且完整
4. 若未配置，则进入 `references/setup-telegram.md` 定义的初始化引导
5. 读取消息模板
6. 结合当前任务结果填入项目名、session 标识、task 短标签、任务开始时间等最小字段
7. 调用发送脚本，把消息发到 Telegram
8. 无论发送成功还是失败，都继续正常向用户汇报主任务结果

## 首次初始化体验

首次需要发送通知时，如果 `config.local.env` 不存在或缺少必要字段，Agent 不要求用户手动研究配置，而是直接进入一次极短的初始化引导。

初始化只收集两个值：

- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`

Agent 在收到后直接写入 `skill/telegram-notifier/config.local.env`，然后回到原本的通知流程继续发送。本次初始化不应阻塞主任务完成；如果用户暂时不想配置，可以跳过本次通知。

## 失败处理

通知能力是附加能力，不能干扰主任务。

因此需要明确以下约束：

- Telegram API 调用失败时，不得导致主任务失败
- 配置缺失时，可以触发初始化引导；若用户暂时不想配置，则跳过本次通知
- 发送脚本应返回清晰的错误原因，便于后续修复
- Agent 在汇报主任务时不需要大篇幅讨论通知失败，除非用户明确询问

## 去重与幂等

本次设计不引入专门的持久化任务状态系统，也不要求写入复杂的去重数据库。

去重要求收敛为行为约束：

- 每个已完成且值得提醒的任务只应调用一次该 skill
- 如果 Agent 在同一轮任务收尾阶段发生重试，应优先避免重复发送
- 若未来使用中发现重复通知问题明显，再补充一个极轻量的本地去重机制

## 安全与隐私边界

消息中只应发送适合出现在 Telegram 的最小提醒信息，不应默认发送敏感内容。

需要明确以下边界：

- 不自动发送密钥、凭证、环境变量或完整输出日志
- 不默认发送大段技术细节或代码内容
- 默认只发送项目名、session 标识、task 短标签、完成状态和任务开始时间

## 验收标准

- 仓库中存在一个定位为全局复用的 `telegram-notifier` skill 目录
- skill 的 `description` 明确说明：当 Agent 判断这次完成值得主动提醒用户时，应调用该 skill
- 文档结构按 `SKILL.md + README-zh.md + references/setup-telegram.md` 分层组织，而不是把全部流程堆在主文件里
- 首次使用时由 Agent 引导完成配置，不要求用户手动研究配置流程
- 本地配置项保持轻量，不包含固定分钟阈值
- 消息模板能稳定生成提醒型通知，而不是报告型通知
- 单条消息默认只包含项目名、session 标识、task 短标签、完成状态和任务开始时间
- 发送失败不会影响主任务本身的完成与汇报
