# Telegram notifier 模板进一步精简设计

## 背景

当前 `skill/telegram-notifier/assets/message-template.md` 只有一行固定模板：`[{{project_name}}] Task {{task_label}} Completed.`。

虽然这已经比早期方案轻很多，但它仍然把两类信息绑死在模板里：

- `task_label` 这样的任务标签字段
- `Completed` 这样的固定后缀语气

用户希望继续收窄模板职责：模板只保留真正稳定、跨场景都成立的 `project` 信息；剩余提醒内容不再由模板硬编码，而是直接指导 Agent 根据当前节点自己填写。

## 目标

- 让消息模板只保留 `project` 信息这一项固定骨架
- 去掉模板中对任务标签和固定完成态文案的绑定
- 让 Agent 根据当前场景自主补全后缀消息
- 继续保持通知简短、单行、提醒型、非敏感，同时允许携带少量关键结果

## 非目标

- 不删除 `assets/message-template.md` 文件
- 不修改发送脚本或配置机制
- 不把 Telegram 通知扩展成完整报告消息
- 不引入新的结构化占位字段

## 方案对比

### 方案 A：模板只保留 project 前缀（推荐）

- 模板形态改成精确的 `"[{{project_name}}]"`
- 由 Agent 在拼接最终消息时补上一个空格，再接一小段符合当前检查点的自然语言
- 优点：既保留统一入口，也把固定文案压到最低
- 缺点：后缀表达会更依赖 Agent 判断，但这正是本次目标

### 方案 B：完全删除模板文件

- 在 `SKILL.md` 中直接要求消息必须带上 project 信息
- 优点：结构最极简
- 缺点：失去一个明确的消息骨架落点，主文件责任会变重

### 方案 C：沿用现有句式，仅去掉 task_label

- 例如保留 `Completed` 一类固定后缀
- 优点：改动最保守
- 缺点：仍然没有把“后缀消息由 Agent 自填”这件事落实到位

本次采用方案 A。

## 设计

### 模板职责

`assets/message-template.md` 只保留项目名前缀，不再承载任务标签或固定状态句式。

这样模板只负责一件事：为所有 Telegram 提醒提供统一的项目识别前缀。

为避免后续实现再次漂移，建议继续沿用当前前缀风格，即保留方括号包裹的项目名形式，而不是把 project 信息改写成新的句式。

前缀与后缀之间的分隔采用单一规则：模板文件本身不保留尾部空格，也不附带新的固定标点；Agent 在拼接最终消息时，必须在项目名前缀后补上一个空格，再接后缀正文。这样既保住“模板只含 project 信息”的边界，也避免把可读性建立在隐蔽尾空格上。

### Skill 职责

`SKILL.md` 中不再要求 Agent 填写固定的 `task_label`，也不再要求套用固定的完成句式。规则改为：

- 先读取极小模板前缀
- 再根据当前检查点补上一小段自然语言
- 这段后缀的目的不是汇报细节，而是明确说明“为什么用户现在应该回来”

### 后缀消息约束

为了避免放开后变得松散，后缀消息仍保留以下边界：

- 简短
- 尽量单行
- 非敏感
- 提醒型而非报告型
- 必须明确说明用户为什么现在应该回来
- 明确当前动作是“可审核”“需确认”或“需回复”等，而不是机械重复“已完成”
- 可以附带极少量关键结果，前提是这些内容能帮助用户判断是否值得立刻回来，而不是展开完整汇报

### 场景表达

`SKILL.md` 应从“固定模板套用”改为“按场景生成后缀消息”的指导方式。

建议保留少量意图示例，例如：

- 已可审核
- 需要确认
- 需要回复

这些示例只用于说明方向，不应再次演变成新的硬编码模板集合。

如果当前节点已经明显是“需要用户回来查看”的完成态，Agent 仍然可以直接写自然的完成提醒；重点是不再把某个固定英文后缀当作唯一合法表达。

用户期望的消息风格更接近“短提醒 + 少量关键结论”。这里的“少量关键结论”是指只补充最有助于用户判断优先级的那一点信息，例如：

- 当前回来要做的动作：review、确认、回复
- 一条最关键的结果：已过审、主测试已通过、只剩最后一步验证

这些内容只能作为提醒的增强信息，不能扩展成阶段性汇报、过程复盘或多句摘要；文档也不应给出可直接套用的整句样板，以免再次演变成软模板。

## 流程调整

新的消息组装顺序为：

1. 读取 `assets/message-template.md` 中的项目名前缀
2. 由 Agent 在前缀后补上一个空格
3. 判断当前是完成、待确认还是待回复等用户回场节点
4. 由 Agent 自行补上一小段贴合该节点的后缀消息
5. 将完整单行消息交给发送脚本

## 验收标准

- `assets/message-template.md` 只保留 project 相关前缀
- `SKILL.md` 不再要求填写 `task_label`
- `SKILL.md` 不再要求使用固定完成态句式
- `SKILL.md` 明确说明后缀消息由 Agent 根据当前检查点自行填写
- `SKILL.md` 明确说明场景示例只是方向，不是新的固定模板集合
- `assets/message-template.md` 本身不依赖尾部空格或固定后缀
- `SKILL.md` 明确规定由 Agent 在拼接时补上前缀后的单个空格
- 后缀正文必须表达用户回场原因，不能只发项目名前缀或空泛状态词
- 后缀正文允许携带少量关键结果，但不能膨胀成多句报告或长摘要
- 整体消息仍保持简短、提醒型、单行优先
