# Telegram notifier 失败后检查配置设计

## 背景

当前 `skill/telegram-notifier/SKILL.md` 要求 Agent 在每次发送前先检查 `config.local.env` 是否存在且完整；如果缺失或不完整，就先进入 setup。

用户希望把这个时机后移：不要在每次发送前主动检查配置，因为这会带来额外动作，也可能引出不必要的权限请求。更合适的方式是先尝试发送，只有在发送失败后，才引导 Agent 去检查配置是否缺失或不完整。

## 目标

- 去掉每次发送前的主动配置检查
- 改为发送失败后再检查配置或进入 setup
- 保持“通知失败不影响主任务完成”的原则不变

## 非目标

- 不修改发送脚本本身
- 不扩展新的配置项
- 不调整提醒文案模板

## 方案对比

### 方案 A：只改 `SKILL.md` 中的规则与流程（推荐）

- 把“缺配置时先 setup”改成“先发送，失败后再检查配置”
- 优点：改动最小，直接改变 Agent 的执行顺序
- 缺点：README 中的描述会暂时比实现口径旧一点，但不影响实际行为

### 方案 B：保留发送前检查，只弱化语气

- 优点：改动保守
- 缺点：仍然没有解决“每次都先检查”的核心问题

## 设计

本次采用方案 A。

### 规则调整

`SKILL.md` 中不再要求 Agent 在发送前检查 `skill/telegram-notifier/config.local.env`。规则改为：

- 正常情况下直接尝试发送
- 只有在发送失败时，才去检查本地配置是否缺失或不完整
- 如果配置确实有问题，再读取 `references/setup-telegram.md` 并完成 setup

### 流程调整

新的流程顺序为：

1. 确认任务已经真正完成
2. 读取消息模板并组装最终提醒内容
3. 直接调用发送脚本尝试发送
4. 如果发送成功，流程结束
5. 如果发送失败，先简短记录失败，再检查 `config.local.env`
6. 只有在配置缺失或不完整时，才进入 setup
7. 如果 setup 成功，则基于同一条最终提醒内容重试一次发送
8. 如果 setup 被跳过、未完成，或重试后仍然失败，则停止补救并继续主任务回复
9. 无论最终是否成功发出提醒，都继续正常完成主任务回复

### 错误处理

发送失败后不应默认认定一定是配置问题，因此文案需要写成“检查配置是否缺失或不完整”，而不是“确认配置错误”。

这样可以兼容：

- 本地未配置
- 配置不完整
- Telegram 网络异常
- Telegram API 暂时不可用

如果失败后发现是配置缺失或不完整，并且 setup 已成功完成，Agent 应只重试一次发送，避免在收尾阶段陷入重复尝试。

如果用户在 setup 中选择暂不配置，或者 setup 没有完成，则直接放弃本次提醒，继续正常结束主任务。

## 验收标准

- `skill/telegram-notifier/SKILL.md` 不再包含“每次发送前先检查配置”的要求
- `skill/telegram-notifier/SKILL.md` 明确写出“发送失败后再检查配置”的触发逻辑
- `skill/telegram-notifier/SKILL.md` 明确写出 setup 成功后仅重试一次发送
- 发送失败时仍然保持 best-effort，不阻塞主任务结束
